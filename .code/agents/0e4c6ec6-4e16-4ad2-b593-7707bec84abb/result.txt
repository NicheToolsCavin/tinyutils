**Vendor Outline**

- **Fetch script**: add `scripts/vendor_pandoc.py` (or `.sh` if preferred, just keep it under `scripts/`). Script responsibilities: download Linux `pandoc-3.1.11.1` tarball from upstream release, verify checksum if available, extract binary, and write it to `tinyutils/api/_vendor/pandoc/pandoc` with `0755` mode. Script intentionally refrains from touching macOS binaries to keep repo thin; mac devs keep using their local `pandoc` installation.

- **Repo layout**: introduce `tinyutils/api/_vendor/` with `.gitkeep` if needed; actual binary ends up at `tinyutils/api/_vendor/pandoc/pandoc`. Ignore other extracted files to stay surgical.

- **Runner wiring (`tinyutils/api/_lib/pandoc_runner.py`)**:  
  - First preference: honor `os.environ.get("PYPANDOC_PANDOC")` so local overrides or future config keep working.  
  - If unset, construct the vendored path (`Path(__file__).resolve().parent.parent / "_vendor" / "pandoc" / "pandoc"`).  
  - If that file exists, pass its string path into the pandoc invocation (either by setting `os.environ["PYPANDOC_PANDOC"]` before calling `pypandoc` or by pointing subprocess calls directly).  
  - If neither env nor vendor binary exists, fall back to current behavior (likely relying on system `pandoc`) so mac devs aren’t blocked.

- **Service layer touch point (`tinyutils/convert/service.py`)**: ensure whatever helper constructs the converter uses the runner’s resolved path—no hard-coded references elsewhere. A brief comment there explaining that Vercel uses the vendored binary helps future maintainers.

- **Tests**: add/adjust a lightweight test (probably under existing service tests) that mocks `pandoc_runner` resolution to confirm it prefers env var, then vendored path, then default. Since we won’t ship the actual binary into CI for the test, use `tmp_path` or monkeypatch `Path.exists` to simulate presence/absence.

- **Local vs preview behavior**:  
  - Local mac devs: normal workflow continues; `PYPANDOC_PANDOC` remains optional. They can run the vendor script to update the Linux binary but don’t need it for day-to-day work.  
  - Vercel preview/serverless: deploy artifact includes `tinyutils/api/_vendor/pandoc/pandoc`; the runner automatically picks it up because system `pandoc` isn’t available. No extra env configuration required in Vercel—just ensure deploy step runs the vendor script before build (or commit the binary if policy allows).  
  - Document in script docstring or README snippet that the vendor script must be run manually when bumping pandoc.

Outcome: single script for reproducible vendoring, explicit runner fallback chain, minimal test coverage proving resolution order, and clear behavior split between developer laptops and serverless previews.
