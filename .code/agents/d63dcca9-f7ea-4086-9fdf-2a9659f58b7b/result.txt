Here is a plan to address your goals:

1.  **Vercel Authentication:** To enable unattended deployments, securely store your Vercel token (`VERCEL_TOKEN`) and project ID (`VERCEL_PROJECT_ID`) as environment variables in your CI/CD environment. This avoids exposing credentials in scripts.

2.  **Reliable Preview Deployment:** Create a deployment script that uses the Vercel CLI (`vercel --prod`) and polls for the deployment status. The script should check the deployment status periodically and wait for a "READY" state before proceeding. This ensures that subsequent steps only run after a successful deployment.

3.  **Smoke Testing:** Develop a Python script using a library like `requests` to perform smoke tests on the deployed preview URL. This script should:
    *   Take the preview URL as an input.
    *   Send test requests to the converter endpoints.
    *   Validate the responses and status codes.
    *   Save the test results (including request/response data) to a timestamped artifact file (e.g., `smoke-test-results-YYYY-MM-DDTHH-MM-SS.json`).

4.  **Artifact Management:** Store the smoke test artifacts in a dedicated directory within your project (e.g., `tests/smoke_artifacts`). This allows for easy tracking and comparison of test results over time.

5.  **Automated Documentation Updates:** Create a script to automatically update your documentation files. This script should:
    *   Read the relevant documentation file (e.g., `DEPLOY_PRODUCTION_CHECKLIST.md`).
    *   Append a new entry with the current Madrid-timestamped date and deployment details.
    *   Use a library like `pytz` to ensure the timestamp is in the correct timezone.

6.  **Rollback and Benchmarking Policy:**
    *   **Rollback:** For preview deployments, a rollback is not typically necessary. If a preview deployment fails, the old preview URL will still be active. For production deployments, you can use the `vercel rollback` command to revert to a previous successful deployment.
    *   **Benchmarking:** After each successful deployment, run a simple benchmark test to measure the response time of the key converter endpoints. Store these results along with the smoke test artifacts to track performance over time.

7.  **Verification and Reporting:** At the end of the deployment process, generate a summary report that includes:
    *   The final deployment URL.
    *   A summary of the smoke test results.
    *   A link to the full smoke test artifact.
    *   The performance benchmark results.

This plan provides a clear path to achieving a reliable, automated deployment and testing process for your universal converter backend.
