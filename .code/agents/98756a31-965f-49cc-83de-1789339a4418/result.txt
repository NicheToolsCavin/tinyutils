Key assumptions / questions

*   **Input `from` parameter:** Assumed that all files in a batch will have the same `from` format, or that the `from` format can be inferred per file if not provided. Clarification needed on how to handle mixed `from` formats in a single batch.
*   **Output `to` parameter:** Will accept an array of strings (e.g., `["md", "html", "txt"]`) or a single string.
*   **`options` parameter:** Will be extended to support per-target options or global options.
*   **Deterministic Metadata:** Implies structured parsing of Pandoc's output or a separate metadata extraction step.
*   **Error Handling:** Errors will be aggregated and reported for batch conversions (per-file errors vs. global batch errors).
*   **Performance:** Batch conversions could be resource-intensive; performance targets or concurrency requirements need to be considered.
*   **Blob Storage:** Existing blob upload contract will be maintained, implying separate blob uploads for each output file.
*   **Lua Filters:** Existing Lua filters are assumed to be compatible with multiple output formats or will need adaptation.

Proposed architecture

A new `tinyutils/convert/` library will orchestrate the conversion process.

*   **`tinyutils/convert/` (New Library):**
    *   `converter.py`: Main orchestration module with `convert_batch` and `_convert_single_file` functions.
    *   `pandoc_processor.py`: Encapsulates Pandoc execution, Lua filter application, and output parsing (evolution of `tinyutils/api/_lib/pandoc_runner.py`).
    *   `output_formatter.py`: Handles post-Pandoc processing for different output formats.
    *   `metadata_extractor.py`: Extracts deterministic metadata (headings, snippets, images).
    *   `blob_manager.py`: Handles interactions with blob storage (abstracting `tinyutils/api/_lib/blob.py`).
*   **`tinyutils/api/convert/index.py` (API Endpoint):** Refactored to use the new `tinyutils/convert/` library, including input validation and JSON envelope construction.
*   **`tinyutils/api/_lib/`:** Existing modules (`text_clean.py`, `manifests.py`, `utils.py`) will be used by the new library.

Testing & evidence strategy

*   **Unit Tests:** For each new module (`pandoc_processor.py`, `output_formatter.py`, `metadata_extractor.py`, `blob_manager.py`, `converter.py`).
*   **Integration Tests:** For `tinyutils/api/convert/index.py` to verify end-to-end flow.
*   **Regression Tests:** New tests under `tinyutils/tests/` covering single-file/multi-target, multi-file/single-target, multi-file/multi-target, and error handling scenarios.
*   **Fixtures:** New directory `tests/fixtures/convert/` with sample input documents and expected output files/metadata.
*   **Artifacts:** Store conversion outputs for successful test runs in `tinyutils/artifacts/convert/<date>/` for evidence and manual inspection.
*   **Tool Versions:** Ensure `toolVersions` in the output envelope accurately reflect external tool versions.

Risk / mitigation list

*   **Risk: Performance Degradation for Batch Conversions.**
    *   **Mitigation:** Implement asynchronous processing, monitor resource usage, consider worker queues.
*   **Risk: Inconsistent Output Across Formats/Files.**
    *   **Mitigation:** Standardize Pandoc arguments/filters, implement robust post-processing, comprehensive regression tests.
*   **Risk: Breaking Existing API Contract.**
    *   **Mitigation:** Maintain backward compatibility for single input/Markdown output, use versioning if necessary.
*   **Risk: Complex Error Reporting for Batches.**
    *   **Mitigation:** Design clear error structure in output envelope, detailed logging.
*   **Risk: Difficulty in Extracting Deterministic Metadata.**
    *   **Mitigation:** Leverage Pandoc's AST output (JSON), use reliable parsing libraries, iterate on `metadata_extractor.py`.
*   **Risk: Increased Complexity and Maintainability Burden.**
    *   **Mitigation:** Adhere to clear module boundaries, well-documented code, comprehensive test suite.
*   **Risk: Security Vulnerabilities (e.g., Pandoc execution).**
    *   **Mitigation:** Run Pandoc in isolated environment, sanitize user inputs.
*   **Risk: Large Blob Storage Consumption.**
    *   **Mitigation:** Implement efficient storage strategies, consider temporary storage and cleanup, effective media extraction zipping.
