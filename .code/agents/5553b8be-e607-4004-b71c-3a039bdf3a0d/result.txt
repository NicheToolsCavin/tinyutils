**Diagnosis**
- Every preview failure shows `FUNCTION_INVOCATION_FAILED` (see `tinyutils/docs/AGENT_RUN_LOG.md:8`) which means the lambda crashed before FastAPI could format an error response. The only unhandled path early in a request is the eager `ensure_pandoc()` call at the top of `convert_one` (`tinyutils/convert/service.py:46-88`). On Vercel that call tries to download a 140 MB tarball from GitHub on cold start (`tinyutils/api/_lib/pandoc_runner.py:16-78`). Network timeouts, tmp quota issues, or blocked outbound traffic raise `requests`/`OSError` exceptions that are **not** wrapped in `PandocError`, so the general `try/except` later never runs and the exception bubbles up to FastAPI as a 500. Local tests pass because pandoc is already present so the eager call never raises.
- When `_download_input` enforces limits via `ensure_within_limits` it raises `JobTooLargeError` (`tinyutils/api/_lib/utils.py:35-80`). The API layer only handles `blob.DownloadError` and turns every other exception (including `JobTooLargeError` or `PermissionError` creating the temp workspace) into a 500 (`tinyutils/api/convert/index.py:94-125`). In a size spike or a `/tmp` permission issue the preview will fail indistinguishably from a server bug, making smoke triage painful.
- Because `pandoc_runner.ensure_pandoc` does not wrap download/extract failures, even after removing the eager call any transient network issue will still raise a raw `requests.RequestException` and skip the markdown fallback path. This keeps the converter brittle in exactly the environment we are targeting (serverless).

**Patch Plan**
1. `tinyutils/convert/service.py` – remove the direct `ensure_pandoc()` call near the top of `convert_one` so that all pandoc provisioning is covered by the existing guarded block. No behaviour change when pandoc is available; when it is not, the converter will drop into `_fallback_conversion` instead of crashing the request.
2. `tinyutils/api/_lib/pandoc_runner.py` – wrap the download/extract logic in `try/except` blocks that re-raise every `requests.RequestException`, `tarfile.TarError`, or `OSError` as `PandocError`. Add a modest download timeout so cold starts fail fast. With this in place any provisioning issue is converted into the same error type `convert_one` already knows how to degrade.
3. `tinyutils/api/convert/index.py` – import `JobTooLargeError` and add dedicated handlers both around `_download_payloads` and around `convert_batch`. Return `413 Payload Too Large` when the size guard triggers, and emit a clearer `500 Temporary storage unavailable` if `/tmp` cannot be created. This directly addresses the “strict validation / tmp permissions” guard-rail request.
4. Tests – extend `tests/convert/test_service.py` with a regression that forces `pandoc_runner.ensure_pandoc` to raise and asserts that we fall back to passthrough output instead of crashing. Extend `tests/convert/test_api_convert.py` with two FastAPI tests that prove we now return 413 when `_download_payloads` or `convert_batch` surfaces `JobTooLargeError`.
