  - Pin `pydantic` and its matching `pydantic-core` version; force binary-only install.
- Vendor wheels (robust on CI)
  - Pre-download platform wheels locally and install from a committed `wheels/` folder.

  - Pin `pydantic<2` and compatible FastAPI; apply small code shims.
- Containerize/alternate runtime (last resort)
  - Deploy via Dockerfile or alternate host to fully control ABI/GLIBC.

**Pre-Checks (quick wins)**
- Vercel runtime: not Edge; use Serverless Python.
- Python version: set to `3.11`.
- Build flags: force wheels.
  - Add envs: `PIP_ONLY_BINARY=:all:`, `PIP_NO_BUILD_ISOLATION=1`.
- Confirm platform wheels exist for your Python: `cp311 manylinux2014_x86_64`.

**Option 1 — Pin Compatible Pydantic v2 + pydantic-core (least invasive)**
- Why: Keeps current code; binary-only avoids Rust builds on Vercel.
- How:
  - Add to `requirements.txt` (example pair; adjust to your current FastAPI):
    - `fastapi==0.111.0`
    - `pydantic==2.7.4`
    - `pydantic-core==2.18.4`
  - Enforce wheels during install:
    - `pip install --only-binary=:all: -r requirements.txt`
    - Or set env: `PIP_ONLY_BINARY=:all:`
- Trade-offs:
  - Pros: Minimal code change, fast.
  - Cons: Must pick a pair that has prebuilt wheels for `cp311` on manylinux2014.

**Option 2 — Vendor Wheels (resilient CI installs)**
- Why: Eliminates network/platform variance; guarantees exact wheel used by Vercel.
- How:
  - Locally, download platform wheels (example):
    - `pip download --only-binary=:all: --platform manylinux2014_x86_64 --python-version 311 --implementation cp -d wheels pydantic==2.7.4 pydantic-core==2.18.4`
  - Commit `wheels/` and install on Vercel with:
    - `pip install --no-index --find-links=./wheels -r requirements.txt`
- Trade-offs:
  - Pros: Very reliable; no compiling on Vercel.
  - Cons: Slight repo bloat; must refresh wheels when upgrading Python/OS or packages.

**Option 3 — Revert to Pydantic v1 (pure Python path)**
- Why: Removes native extension dependency entirely.
- How:
  - Pin:
    - `fastapi==0.109.*` (last series with v1)

    - `model_validate(...)` → `parse_obj(...)`

    - Replace `RootModel[T]` with a wrapper model or field.
  - Re-run FastAPI validation paths and update any `json_schema` calls.
- Trade-offs:
  - Pros: No compiled deps; simplest runtime story.
  - Cons: Backwards migration may touch multiple models; slower than v2; diverges from current FastAPI if you’re already on 0.110+.

**Option 4 — Containerize or Alternate Host (most invasive)**
- Why: Full control of OS/GLIBC/Python, sidesteps Vercel binary availability.
- How:
  - Provide a `Dockerfile` (Python 3.11 slim + `pip install`) and deploy as a container workload (Vercel with Dockerfile or another platform).
- Trade-offs:
  - Pros: Maximum control and reproducibility.
  - Cons: Larger cold starts, higher complexity/cost vs. serverless.
