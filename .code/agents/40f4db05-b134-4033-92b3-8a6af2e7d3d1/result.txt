[2025-11-10T03:56:51] codex

**Pinning Strategy**

- Use `vercel.json` per-function override: place at repo root with `"functions": { "api/convert.py": { "runtime": "python3.11" } }`. File path must match deployed file (e.g., `api/convert/index.py` → `"api/convert/index.py"`). This is the only Vercel-supported way to pin runtime per function; spelling must follow camelCase keys and valid runtime string.
- If all Python endpoints should be 3.11, set the default runtime in Vercel Project Settings → Functions → Default Runtime = Python 3.11. CLI `vercel env pull`/project JSON doesn’t change runtimes—only `vercel.json` or the dashboard applies.
- Environment variables are unrelated; do not rely on `PYTHON_VERSION`—Vercel ignores it for runtimes.

**Verification After Deploy**

- After `vercel deploy`, open the deployment’s build output: logs show `Function Runtime: python3.11` per function.
- Download `.vercel/output/config.json` (formerly `.vc-config`) from the deployment artifact or via `vercel pull` and confirm the `runtime` entry under the matching function is `python3.11`.
- Hit the API endpoint once; the function log in Vercel dashboard should list `Runtime: python3.11` in the invocation metadata.

**Troubleshooting if 3.12 Appears**

- File mismatch: ensure the key in `vercel.json` exactly matches the compiled entry file (`api/convert.py` vs `api/convert/index.py` depending on structure).
- Missing repo root: `vercel.json` must live at project root committed before deploy; redeploy after changes.
- Conflicting project default: dashboard default runtime set to 3.12 overrides functions without explicit entries—make sure your function is listed.
- Cached build: run `vercel deploy --force` to bypass cached build artifacts that may have been produced before the config change.
- Wrong framework preset: if using Vercel monorepo/project settings, verify the correct project directory is selected so `vercel.json` is picked up.
