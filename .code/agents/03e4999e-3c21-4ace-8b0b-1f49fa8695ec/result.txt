**Pattern Summary**
- Edge handlers consistently declare `export const config = { runtime: 'edge' }` and expose `export default async function handler(req)` (e.g., `api/check.js:1`, `api/fence.js:47`), force `cache-control: no-store`, attach `x-request-id`, and return JSON envelopes with either `{ ok: true, … }` or `{ error/code/meta }` using small helper functions ( `api/check.js:7-33`, `api/wayback-fixer.js:7-25` ).
- They almost always require `POST` (except `health`) and implement their own parsing/validation plus concurrency caps, using browser-compatible `fetch` + `AbortSignal.timeout` patterns, so new endpoints should mirror that structure for consistency.
- Python serverless functions live under `api/<tool>/index.py` or `api/<tool>/<action>.py`, spin up `FastAPI()` instances, lean on `_lib` for blob IO, size limits, logging, and respond with deterministic dicts containing `jobId`, file manifests, and diagnostic logs (see `api/convert/index.py:75-162`, `api/findreplace/apply.py:87-172`).
- Secrets & config currently surface through environment variables (`PREVIEW_SECRET` in `api/fence.js:53`, blob/size limits in `_lib`), so any new endpoint should follow the same `process.env` / `os.getenv` pattern and avoid embedding secrets in code.
- Overall JSON schemas favor top-level meta blocks plus arrays of work items (rows/results/outputs). Reusing `_lib` helpers (for downloads, workspaces, logs, diff limits, etc.) keeps behavior consistent between new Python endpoints and existing ones.
